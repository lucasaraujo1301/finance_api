# AGENTS.md — Project Architecture Guide

This file guides AI agents (OpenCode and similar tools) on the structure,
conventions, and rules of this project. Read it before creating, moving, or
editing any file.

## 1. Overview

FastAPI project organized into **vertical modules** (feature-based), with a
`core` module containing the shared base used by all other modules.

```
my_project/
├── migrations/          # Generated by Alembic — do not edit manually
├── modules/
│   ├── core/            # Base: app, config, database, middleware, base model
│   └── <module_name>/   # One module per domain/feature (users, orders, ...)
└── main.py               # Creates and mounts the FastAPI application
```

## 2. Required structure for a module

Every new module created under `modules/<name>/` **must follow exactly**
this skeleton (even if some files start out empty/minimal):

```
modules/<name>/
├── tests/
│   ├── __init__.py
│   └── ...              # integration, service, repository tests, etc.
├── router.py             # Endpoints (APIRouter) — must not contain business logic
├── models.py             # ORM models (SQLAlchemy), inherit from the Base in core
├── services.py            # Business logic / orchestration
├── repository.py         # Data access (queries), isolates the ORM
├── schemas.py             # Pydantic (request/response)
└── dependencies.py       # Depends() providers (dependency injection)
```

Responsibility of each layer:

| File | Can import from | Must not contain |
|---|---|---|
| `router.py` | `schemas`, `dependencies`, `services` | SQL queries, business logic |
| `services.py` | `repository`, `schemas`, `dependencies`, `core` | ORM/SQL details |
| `repository.py` | `models`, `core.database`, `schemas` | business logic |
| `schemas.py` | `core` (base types) | logic |
| `models.py` | `core.models` (Base) | business logic |
| `dependencies.py` | `repository`, `services`, `core` | endpoints |

`core/` provides only cross-cutting primitives: `config.py`, `database.py`
(engine/session), `models.py` (declarative Base), `middleware/`. No feature
module should ever be imported inside `core/`.

## 3. Checklist when creating a new module

1. Create the `modules/<name>/` folder with all the files from section 2,
   even if some start out as just a placeholder/`pass`.
2. Create `modules/<name>/tests/__init__.py` and replicate the test patterns
   already used in `modules/users/tests/` (fixtures, naming, reuse of
   `conftest.py` inherited from `core/tests/conftest.py`).
3. Register the new module's `router.py` in `main.py`, following the same
   inclusion pattern used for `users`.
4. If a new table is needed, create the model in `models.py` inheriting from
   the Base in `core/models.py`, then generate the migration via Alembic
   (`alembic revision --autogenerate`) — never edit `migrations/` by hand.
5. Never duplicate infrastructure logic (pagination, authentication,
   logging) — that belongs in `core/`.

## 4. Inter-module dependency rule (anti-cycle)

**Allowed direction:** `modules/<feature>` → `modules/core`. Never the
reverse.

**Between feature modules (`users`, `orders`, etc.):**

- A module **must not directly import** `router.py`, `services.py`,
  `repository.py`, or `models.py` from another feature module.
- If module A needs data/behavior from module B:
  - Prefer exposing a small, stable interface in B (e.g., a function in B's
    `services.py`, or a public schema in B's `schemas.py`) and import
    **only that** in A — never the reverse at the same time (that would
    already be a cycle).
  - If A and B genuinely need to reference each other, extract the shared
    part into `core/` (or a new shared module, e.g. `modules/shared/`), so
    both depend only on it.
  - For cross-module foreign keys, reference the table by string
    (`ForeignKey("orders.id")`) instead of importing the other module's
    model class, avoiding a physical cross-import.
- Prefer communication between modules via **events/callbacks** registered
  in `core` (e.g. a simple event bus) when the relationship is inherently
  bidirectional.

**Before opening an import of `modules/X` inside `modules/Y`, the agent must
check:** does `modules/X` already import (directly or indirectly) something
from `modules/Y`? If so, **do not add the import** — refactor into `core/`
or into a one-directional interface instead.

## 5. Tests

- `core` tests live in `modules/core/tests/`, with reusable fixtures in
  `modules/core/tests/fixtures/` (e.g. `db_session`, `client`) exposed via
  `conftest.py`.
- Each feature module's tests live in `modules/<name>/tests/` and reuse
  `core`'s fixtures via `conftest.py` — do not recreate `db_session`/`client`
  locally.
- Suggested naming: `test_router_*.py`, `test_service_*.py`,
  `test_repository_*.py`, mirroring the files of the layer being tested.

## 6. When generating code, the agent should

1. Locate the module corresponding to the task's domain; if it doesn't
   exist, create it following sections 2 and 3.
2. Place each piece of code in the file for the correct layer (table in
   section 2).
3. Mentally run the check from section 4 before adding any import between
   feature modules.
4. Keep `main.py` as the single place that knows about all routers.
5. Never edit files inside `migrations/` manually.
